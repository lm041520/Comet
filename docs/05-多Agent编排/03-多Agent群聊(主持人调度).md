# 多 Agent 群聊（主持人调度）— 设计与面试

> 2~5 个角色卡 + 真人在一个群里聊天，由「主持人 LLM」调度谁发言、按序接话，像微信群。
> 对应能力域：**多 Agent 编排 / 多角色协作**。代码：`core/agent/group_chat.py`（主持人调度 + 接话）+ `group_chat_service.py` + `prompts/group_host.jinja2` / `group_speaker.jinja2`。

---

## 0. 能力定位（对应招聘要求）

- 对应 JD：**「Multi-Agent 多智能体协作」「Agent 调度 / 编排」「角色扮演」**。
- 角色：项目里最能体现「多 Agent 协作」的特性——多个 AI 角色 + 真人同群，需要解决「谁该说话、怎么接话、怎么承载多方上下文」。

---

## 1. 解决什么问题

- **痛点**：单聊是「用户↔一个 AI」。想要「多个不同人设的 AI + 真人」在一个群里自然对话——但这带来三个新问题：① 一条消息后该哪个角色接话？② 角色怎么"看到"别人（其他 AI、其他真人）说的话？③ 怎么避免所有角色同时抢着回、或答非所问。

---

## 2. 数据流

```mermaid
flowchart TD
  MSG[用户发言/@某角色] --> MENTION{有@指定?}
  MENTION -->|是| ONE[只让被@角色回 跳过主持人]
  MENTION -->|否| HOST[主持人LLM decide_speakers]
  HOST -->|返回发言顺序| ORDER[角色名列表 或 空=不接话]
  ORDER --> LOOP[按序逐个发言]
  LOOP --> SPK[stream_speaker 角色流式]
  SPK -->|本轮transcript动态累加| LOOP
  LOOP --> DONE[逐角色头像气泡冒泡前端]
```

## 3. 核心设计与实现（后端）

### 3.1 上下文用「文本 transcript」承载多方对话（关键设计）

单聊上下文是标准的 `[SystemMessage, HumanMessage, AIMessage...]`。但群聊里，对**角色 A** 而言，**角色 B 说的话**既不是自己（不能当 AIMessage）也不是用户（不能当 HumanMessage）。所以群聊**不用消息角色数组，改用带发言人前缀的文本 transcript**：
```
【用户】今天股票怎么样
【小美】大盘还行
【老王】我觉得科技股要调整
```
`build_transcript` 把多方历史渲染成这种文本，作为**场景信息**整段放进每个角色的 system prompt。这是承载「多方对话」最稳的方式。

> 面试一句话：群聊上下文不用 system/human/ai 消息数组——因为对某角色而言别人的话既非自己也非用户。改用「带发言人前缀的文本 transcript」整段作为场景呈现，让每个角色看清谁说了啥。

### 3.2 主持人 LLM 调度发言顺序（`decide_speakers`）

每轮先调一次「主持人」LLM 决定谁发言：
- 喂给主持人：各角色的人设简介（截断）+ 最近几条对话 + 用户最新发言。
- 主持人返回**发言顺序角色名列表**：非空=这些角色按序发言；**空列表=本轮 AI 不接话**（比如两个真人在互相聊，AI 不必插嘴）。
- **解析失败/异常兜底返回全员**，保证对话不中断。

这解决了「谁该说话」——不是所有角色都抢着回，而是主持人按对话情境决定。

### 3.3 @ 指定跳过主持人（`parse_mention`）

用户消息里 `@某角色` 时，直接只让被 @ 的角色回，**跳过主持人调度**（用户明确指定了就别再让主持人判断）。

### 3.4 transcript 一轮内动态累加 = 接话（核心技巧）

主持人定了「小美→老王」的顺序后，**逐个发言，且 transcript 在一轮内动态累加**：小美发完，把小美这句加进 transcript，再让老王基于「包含小美刚说的话」的 transcript 发言。于是老王能**接住小美刚说的话**（"同意小美说的…"），形成真正的对话而非各说各的。

> 面试一句话：多角色按主持人定的顺序逐个发言，每个角色发完就把它的话累加进 transcript 再让下一个角色发——这样后面的角色能看到并接住前面角色刚说的话，实现「接话」而非各自独立回答。

### 3.5 角色发言的 system prompt（`build_speaker_messages`）

每个角色发言时拼：角色人设 + 群聊场景说明（`group_speaker.jinja2`：怎么区分发言人、别每句 @、别复述别人、有自己态度）+ 当前 transcript + 当前日期。真人模式叠加口语风格（见情绪篇）。因为有些 provider 不接受「只有 system 无 user」的消息，末尾补一条「现在轮到你发言」的 HumanMessage 触发。

### 3.6 进阶：群级工具 + 多模态 + 可分享

- **群级工具开关**（enable_tools）：开启则每个角色发言走单聊的工具编排（function calling/ReAct），能查知识库/记忆/联网/MCP。
- **群聊带图**：每角色用多模态模型看图发言；带图 + 工具 + 支持 function calling 时能边看图边调工具（如发股票图各角色联网分析）。
- **可分享**：复用对话分享的快照机制，扩展支持多发言人（sender_name/sender_avatar）。

## 4. 关键设计取舍

| 决策点 | 选了什么 | 备选 | 为什么 |
|--------|---------|------|--------|
| 多方上下文 | 文本 transcript 带发言人前缀 | 消息角色数组 | 别人的话既非自己也非用户，文本承载最稳 |
| 谁发言 | 主持人 LLM 调度 | 全员都回 / 轮流 | 按情境决定谁接话，更自然、不刷屏 |
| @ 指定 | 跳过主持人直接回 | 仍走主持人 | 用户明确指定了不必再判断 |
| 接话 | transcript 一轮内动态累加 | 各角色独立回 | 后发言角色能接住前面的话，真对话 |
| 空发言 | 主持人可返回空(不接话) | 必须有人回 | 真人互聊时 AI 不必插嘴 |
| 调度失败 | 兜底全员发言 | 报错 | 保证对话不中断 |

---

## 5. 踩坑与解决

- **角色把别人的话当自己/用户的说**：解法：transcript 带发言人前缀，prompt 教怎么区分发言人。
- **所有角色抢着回、刷屏**：解法：主持人 LLM 按情境决定发言顺序，可返回空不接话。
- **角色各说各的不接话**：解法：transcript 一轮内动态累加，后发言者看到前面的话。
- **只有 system 无 user 报错**（智谱/通义）：解法：末尾补一条 HumanMessage 触发发言。
- **角色每句开头 @ 别人**：解法：prompt 明确「别动不动 @」。

## 6. 面试问答

**Q1（核心）：多 Agent 群聊怎么编排的？**
2~5 个角色卡 + 真人同群。每轮先调主持人 LLM 决定谁发言（按对话情境，可返回空表示 AI 不接话），@ 指定则跳过主持人直接让被 @ 角色回；角色按序逐个流式发言，transcript 一轮内动态累加让后面角色接住前面的话。

**Q2（关键）：群聊上下文怎么组织？为什么不用消息数组？**
不用 system/human/ai 消息数组——对某角色而言，别人（其他 AI/真人）说的话既不是自己也不是用户，没法归类。改用带发言人前缀的文本 transcript（【小美】…【老王】…）整段作为场景信息放进 system prompt，让角色看清谁说了啥。

**Q3（核心）：怎么实现"接话"而不是各说各的？**
主持人定了发言顺序后逐个发言，每个角色发完就把它的话累加进 transcript，再让下一个角色基于更新后的 transcript 发言。这样后面的角色能看到并接住前面角色刚说的话，形成真对话。

**Q4（设计）：主持人调度返回空列表代表什么？**
本轮 AI 不接话。比如群里两个真人在互相聊，不需要 AI 插嘴。让对话更自然，不是每条消息都强行让 AI 回。

**Q5（健壮性）：主持人调度失败怎么办？**
兜底返回全体成员发言，保证对话不中断。宁可全员回也不让群聊卡死。

**Q6（进阶）：群聊能用工具/看图吗？**
能。群级工具开关开启则每角色发言走单聊的工具编排（function calling/ReAct）查知识库/记忆/联网/MCP；带图时每角色多模态看图发言，支持 function calling 时还能边看图边调工具。

## 7. 相关论文 / 概念

**① 多智能体协作（Multi-Agent Systems）**
让多个 LLM Agent 协作完成任务是 2023 起的热点方向：
- **Generative Agents（斯坦福小镇，Park et al. 2023）**：多个 Agent 在虚拟环境自主交互、对话、形成社会行为。本项目群聊是「多角色对话」的轻量版。
- **CAMEL（Li et al. 2023）**：两个 Agent 角色扮演协作完成任务（如 AI 用户 + AI 助手）。
- **AutoGen（Microsoft 2023）**：多 Agent 对话框架，支持 GroupChat + GroupChatManager——**一个 manager 决定下一个发言者**。本项目的「主持人 LLM 调度发言顺序」与 AutoGen GroupChatManager 思路一致。
- **MetaGPT / ChatDev**：给 Agent 分配角色（产品/开发/测试）模拟软件公司协作。

**② 发言者选择（Speaker Selection）**
多 Agent 群聊的核心问题就是「下一个谁说」。策略有：轮询（round-robin）、规则路由、**LLM 自主选择（manager/主持人决定）**。本项目用主持人 LLM 按情境选，并支持 @ 指定和「不接话」，比固定轮询自然。

**③ 角色扮演与人设一致性**
每个角色靠 system prompt 维持人设（见角色卡篇）。群聊里还要让角色「知道自己是谁、别人是谁、别替别人说话」——靠 transcript 发言人标注 + prompt 约束。

**④ 对话上下文的表示**
多方对话用「带说话人标签的文本」表示是对话系统的常见做法（比强行套二元 user/assistant 更自然）。本项目 transcript 即此。

> 一句话脉络：多智能体协作从 Generative Agents/CAMEL/AutoGen 发展而来，群聊的核心是「发言者选择」——本项目用主持人 LLM 调度（同 AutoGen GroupChatManager 思路）+ @ 指定 + 可不接话，用带发言人前缀的 transcript 承载多方上下文 + 动态累加实现接话。
