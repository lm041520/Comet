# 实体去重与图混合检索 — 设计与面试

> 萃取写图前怎么避免重复实体（消歧）、检索时怎么从图里找出相关记忆并带出关系。
> 对应能力域：**记忆 / 知识图谱**。代码：`core/memory/extraction/dedup.py`（两层去重）+ `core/memory/retrieval/searcher.py`（图混合检索）。

---

## 0. 能力定位（对应招聘要求）

- 对应 JD：**「实体消歧 / 实体对齐」「知识图谱构建」「向量检索 + 图检索」「Neo4j」**。
- 角色：去重保证图谱**不长重复节点**（同一个「用户」只有一个），检索保证问答能**找到对的记忆并带出关联事实**。

---

## 1. 解决什么问题

- **去重痛点**：同一实体（「用户」「腾讯」）会在多次萃取里反复出现，不去重图里就一堆重复节点，关系散落、检索失真。
- **检索痛点**：怎么从图里找到和问题相关的实体？光找到实体不够，还要把它的**关联关系**（用户—养→多多）一起带出来给 LLM。

## 3. 核心设计与实现（后端）

### 3.1 为什么去重要「规则初筛 + LLM 精判」

全量两两问 LLM「这俩是不是同一个」太慢太贵（O(n²) 次 LLM 调用）。所以分两步：
- **规则初筛**（廉价）：用文本相似度（`difflib` 序列匹配）、向量余弦、包含关系，筛出**可疑候选对**——只有相似度超过门槛（0.8）或有包含关系的对才进下一步。
- **LLM 精判**（贵但准）：只对可疑候选对调 LLM 判定是否同一实体，置信度 ≥ 0.8 才合并。
把 LLM 调用量从「所有对」降到「少数可疑对」，兼顾效率和准确。

> 面试一句话：实体消歧不全量问 LLM，先用文本/向量相似度 + 包含关系廉价初筛出可疑候选对，只对可疑对调 LLM 精判，把 O(n²) 的 LLM 调用降到很少。

### 3.2 第一层：批内去重（`dedup_within_batch`）

本次萃取出的实体之间先合并：
1. **同名同类型直接合并**（不问 LLM，快路径）：归一化名字相同 + 类型相同就直接并，记进重定向表。
2. **相似候选对问 LLM**：剩下的两两比，满足「名称文本相似 / 向量相似 / 包含关系」之一才调 LLM 判定，合并时把别名、描述、动力学属性（重要度取大、提及次数累加）并入保留方。
产出去重后实体集 + **重定向表 redirect1**（被合并实体旧 id → 保留 id）。

### 3.3 第二层：与图谱已有实体融合（`merge_with_graph`）

去重后的实体再和 Neo4j 里**同类型的已有实体**比对：
1. **同名同类型直接复用已有图节点 id**（快路径，不问 LLM）——这是关键修复点（见踩坑）。
2. 否则找最相似的已有实体，超门槛才调 LLM 判定，命中则复用已有 id、合并别名描述。
按类型缓存已有实体列表减少查询。产出 **重定向表 redirect2**（本次实体 id → 已有实体 id）。

两层重定向表合起来，就能把萃取阶段用「局部 id」占位的三元组/mention/事件，翻译成最终落库的真实实体 id（见萃取篇 3.6）。

### 3.4 检索：向量 + 全文 + 重要度融合（`search_memory`）

和 RAG 混合检索同源思路，但融合公式多了记忆动力学：
1. **向量召回**：query 向量化，走 Neo4j 向量索引召回相似实体（支持复用外部已算好的 query_vector 省一次 embedding）。
2. **全文召回**：Neo4j cjk 全文索引（中文分词）召回。
3. **融合**：`0.55·向量 + 0.30·全文 + 0.15·重要度`，**长期记忆额外 +0.05**——让「更相关 + 更重要 + 更稳定」的记忆优先浮现。

> 面试一句话：记忆检索是向量召回 + 全文召回归一化融合，但比纯 RAG 多了「重要度加权 + 长期记忆加成」——让高价值、稳定的记忆排前面，这又反哺巩固判定。

### 3.5 检索的图特性：一跳邻居遍历（区别于 RAG）

这是图检索相对文本 RAG 的核心优势。命中实体后，**顺着它的 RELATION 关系边取一跳邻居**（`get_entity_neighbors`），拼成「实体 + 关联事实」：不只返回「用户」这个实体，还带出「用户—养→多多」「用户—在→腾讯」这些关系。`format_memory_context` 把它拼成结构化文本给问答 Agent 的记忆工具。文本 RAG 只能返回命中的文本块，图检索能带出**结构化的关联知识**。

### 3.6 命中回写（反哺巩固）

检索命中后 `bump_entity_access` 给命中实体 `access_count +1` + 更新最近访问时间（失败不影响检索）。这个访问统计是记忆巩固（短期→长期）的判据之一——**越常被检索到的记忆越容易被巩固为长期**，形成「高价值记忆浮现」的正反馈（详见分层巩固篇）。

## 4. 关键设计取舍

| 决策点 | 选了什么 | 备选 | 为什么 |
|--------|---------|------|--------|
| 去重判定 | 规则初筛 + LLM 精判 | 全量问 LLM / 纯规则 | O(n²) 降到少数对，兼顾效率准确 |
| 同名处理 | 直接合并/复用快路径 | 都问 LLM | 防 LLM 非确定性把同名判为不同建重复节点 |
| 两层去重 | 批内 + 图侧 | 只做一层 | 批内防本次重复，图侧防跨萃取重复 |
| 检索融合 | 向量+全文+重要度+长期加成 | 纯向量 | 高价值稳定记忆优先浮现 |
| 图检索特性 | 一跳邻居关系遍历 | 只返实体 | 带出结构化关联事实，图库优势 |
| 命中回写 | access_count +1 | 不记 | 反哺巩固判定，形成价值浮现 |

## 5. 踩坑与解决

- **图里出现多个「用户」节点**（核心踩坑）：根因是第二层融合即使同名也每次问 LLM，而 LLM 非确定性 + prompt 倾向「拿不准不合并」，导致「用户」等稳定自指实体跨多次萃取被反复判为不同 → 重复建节点。**解法**：加同名同类型快路径（同名直接复用已有 id，不问 LLM）+ 提供历史重复合并接口（按同名同类型分组，把重复节点的边改接到保留节点后删除）。
- **全量问 LLM 去重太慢太贵**：解法：规则初筛只对可疑对问 LLM。
- **向量召回失败整个检索挂**：解法：向量/全文各自 try/except，一路失败降级用另一路。
- **重复 embedding 浪费**：解法：检索支持复用外部已算好的 query_vector（主动召回里 embedding 一次两路复用）。

## 6. 面试问答

**Q1（核心）：实体去重/消歧怎么做的？**
两层 + 「规则初筛 + LLM 精判」。批内去重（本次实体之间）+ 图侧融合（和图里已有）。同名同类型直接合并/复用（快路径不问 LLM），相似的（文本/向量相似度超 0.8 或包含关系）才调 LLM 判定，置信度够才合并。把 LLM 调用从 O(n²) 降到少数可疑对。

**Q2（踩坑）：为什么会出现重复的「用户」节点？怎么修的？**
第二层融合早期即使同名也每次问 LLM，LLM 非确定性 + 倾向不合并，导致「用户」反复被判为不同建重复节点。修复是加同名同类型快路径（同名直接复用已有 id 不问 LLM），并提供历史重复合并接口把重复节点的边改接到保留节点后删除。

**Q3（设计）：记忆检索和 RAG 检索区别？**
都用向量+全文融合，但记忆检索多两点：融合公式加了重要度加权 + 长期记忆加成（高价值稳定记忆优先）；命中实体后做一跳邻居关系遍历，带出结构化关联事实（用户—养→多多），而 RAG 只能返文本块。

**Q4（原理）：一跳邻居遍历是什么？为什么是图库优势？**
命中实体后顺着它的关系边取直接相连的实体和关系，拼成「实体+关联事实」。这正是图数据库擅长的——关系遍历，关系库要多次自连接。让 LLM 拿到的不是孤立实体而是结构化关联知识。

**Q5（进阶）：检索命中回写有什么用？会不会正反馈失控？**
命中给实体 access_count +1，作为巩固判据——常被检索的记忆易被巩固为长期、排序更靠前，形成「高价值浮现」正反馈。为防马太效应淹没新记忆，重要度只占 0.15 权重不主导，新记忆仍能靠向量/全文相关度直接召回。

**Q6（细节）：去重的相似度怎么算的？**
文本相似度用 difflib 序列匹配比，向量相似度用 name_embedding 算余弦，再加包含关系判断。三者取最大超 0.8 门槛或有包含关系才进 LLM 判定。

## 7. 相关论文 / 概念

**① 实体消歧 / 实体对齐（Entity Resolution / Disambiguation）**
判断「不同提及是否指同一实体」是知识图谱、数据集成的核心难题（也叫 record linkage、去重 dedup）。经典做法是「**阻塞（blocking）+ 匹配（matching）**」两阶段——先用廉价规则把可能相同的分到一组（减少比较量），再对组内做精细匹配。本项目的「规则初筛候选对 + LLM 精判」正是这个经典范式的 LLM 版：相似度初筛 = blocking，LLM 判定 = matching。

**② 实体链接（Entity Linking）**
把文本中的实体提及链接到知识库中已有实体（消歧到唯一 id）。本项目第二层「与图谱已有实体融合」就是实体链接——把本次抽的实体链到图里已存在的节点，而非每次新建。

**③ 图检索 vs 向量检索**
向量检索找「语义相似的内容块」，图检索能做「关系遍历 / 多跳推理」。GraphRAG（微软 2024）的核心洞察就是：很多问题需要的不是「最相似的块」，而是「实体之间的关联」。本项目的「召回实体 + 一跳邻居」是轻量图检索——召回靠向量/全文，关联靠图遍历，两者结合。

**④ 混合检索的融合**
和 RAG 篇同源（归一化加权融合）。本项目记忆检索在此基础上引入「重要度 + 记忆层级」加权，是把「记忆动力学」融入排序的设计（见分层巩固篇）。

> 一句话脉络：实体消歧是 KG 构建核心难题，经典「阻塞+匹配」两阶段对应本项目「规则初筛+LLM 精判」；实体链接把提及链到已有节点（第二层融合）；图检索相对向量检索的优势是关系遍历，本项目「召回实体+一跳邻居」是轻量 GraphRAG。
