# 配置加密 · 连接池 · 统一响应与异常 — 设计与面试

> LLM 应用的工程底座三件套：API Key 用 Fernet 真加密；四存储连接池显式配置；接口统一 `{code,message,data}` + BizError 全局异常。
> 对应能力域：**LLM 应用工程 / 后端工程素养**。代码：`core/security.py`、`db/postgres.py`、`core/response.py`、`core/exceptions.py`。

## 1. 解决什么问题

- **Key 安全**：用户的大模型 API Key 是钱和权限，明文存库一旦泄库就全暴露。
- **连接稳定**：四存储（PG/ES/Neo4j/Redis）高并发下连接耗尽、失效连接、慢查询挂死，要靠连接池参数兜住。
- **接口一致**：前端要可预测的响应结构和明确的中文错误，不能各接口各样、把英文异常栈直接抛出去。

## 3. 核心设计与实现（后端）

### 3.1 API Key 加密：Fernet 对称加密（`security.py`）

- **入库**：`encrypt_secret(plain)` 用 Fernet（基于 AES-128-CBC + HMAC）加密成密文存 `api_key_encrypted` 字段；**绝不存明文**。
- **调用时解密**：resolver/工厂构建客户端时 `decrypt_secret(token)` 还原明文用于请求。
- **接口返回掩码**：`mask_secret` 只露尾 4 位（`****1234`），前端展示「已配置」但拿不到完整 Key。
- 密钥 `FERNET_KEY` 放 `.env`，一次生成长期不变，**绝不进 git**。

> 面试一句话：Key 用 Fernet 对称加密入库、调用时才解密、接口只返回尾 4 位掩码，泄库也拿不到明文；密钥在环境变量里不进代码库。为什么不用哈希？哈希不可逆，但 Key 调用时需要还原明文，所以用可逆的对称加密而非哈希。

**密码则相反用 bcrypt 哈希**（`hash_password`/`verify_password`）——密码只需验证不需还原，用不可逆哈希更安全。**「能还原的用加密、只验证的用哈希」**是这里的关键区分。

### 3.2 JWT 双 token（`security.py`）

`create_access_token`（短期）/ `create_refresh_token`（长期），payload 带 `type` 区分 access/refresh，用 `jwt_secret` HS256 签名。`decode_token` 校验失败返回 None。access 过期用 refresh 换新，避免频繁重登。（详见工程化篇账号鉴权。）

### 3.3 连接池显式配置（`db/postgres.py`）

`create_async_engine` 显式配六个参数，不吃默认值：
- `pool_size` / `max_overflow`：常驻连接数 + 峰值溢出连接数，控并发上限。
- `pool_timeout`：取不到连接的等待超时（防无限等）。
- **`pool_recycle`**：连接存活上限（1800s），防被 DB/中间件单方面断开后拿到死连接。
- **`pool_pre_ping=True`**：取连接前先 ping 一下，**剔除失效连接**——解决「数据库重启/网络抖动后第一次请求必报连接错」的经典问题。
- `connect_args.server_settings.statement_timeout`：asyncpg 层设**单条 SQL 超时**（毫秒），防慢查询挂死连接池。

ES/Neo4j/Redis 同样显式配了 max_connections/pool_size/timeout。`SessionLocal = async_sessionmaker(expire_on_commit=False)`——关闭提交后过期，避免提交后访问对象触发额外查询（异步下尤其重要）。另有 `create_task_engine()` 给 Celery 任务用独立 NullPool 引擎（任务用新事件循环，全局池会绑到旧循环报错）。

### 3.4 统一响应（`response.py`）

所有接口返回 `{ code, message, data }`：成功 `success(data, msg)` → `code:0`；前端用统一拦截器解包 data、按 code 判错。结构固定，前端可预测。

### 3.5 业务异常 + 全局处理器（`exceptions.py`）

- **`BizError(message, code, status_code)`**：业务失败抛它，带**中文提示** + 业务码 + HTTP 码。service 层 `raise BizError("未配置对话模型", code=2010)`，不把英文异常栈抛给前端。
- **全局处理器**（`register_exception_handlers`）：捕获 BizError → 按 status_code 返回统一结构；捕获 `RequestValidationError` → 422「参数校验失败」+ 错误详情；捕获 HTTPException → 统一结构。**前端永远收到 `{code,message,data}`**，不会见到原始堆栈。

> 面试一句话：业务失败统一抛 BizError（中文提示+业务码+HTTP码），全局异常处理器把它和参数校验错、HTTP 错都转成统一的 `{code,message,data}`，前端拿到的永远是可预测结构和中文提示，不暴露异常栈。

## 4. 关键设计取舍

| 决策点 | 选了什么 | 备选 | 为什么 |
|--------|---------|------|--------|
| API Key 存储 | Fernet 对称加密 | base64 伪加密 / 明文 | 可逆（调用要还原明文）+ 真加密；base64 不是加密 |
| 密码存储 | bcrypt 哈希 | 加密 | 密码只验证不还原，不可逆哈希更安全 |
| 连接池 pre_ping | 开启 | 关闭 | 剔除失效连接，避免 DB 重启后首请求报错 |
| SQL 超时 | asyncpg statement_timeout | 应用层超时 | DB 层强制，慢查询不会挂死连接池 |
| 错误返回 | BizError + 全局处理器 | 各接口自己 try/except | 统一中文提示和结构，不暴露异常栈，少重复 |
| 响应结构 | 固定 {code,message,data} | 直接返数据 | 前端可预测、统一拦截解包与错误处理 |

## 5. 踩坑与解决

- **DB 重启后第一次请求必报连接错**：池里是失效连接。解法：`pool_pre_ping=True` 取连接前 ping 剔除。
- **Celery 任务里用全局引擎报「事件循环已关闭」**：全局池绑在主循环，任务用新循环。解法：`create_task_engine()` 给任务用独立 NullPool 引擎。
- **慢查询拖垮连接池**：解法：asyncpg `statement_timeout` 单条 SQL 超时兜底。
- **异步下提交后访问对象触发隐式查询**：解法：`expire_on_commit=False`。
- **alembic.ini 含中文 Windows 报错**：解法：保持纯 ASCII。

## 6. 面试问答

**Q1（安全）：API Key 怎么存的？为什么不用哈希？**
Fernet 对称加密存库，调用时解密、接口返回尾 4 位掩码，密钥在环境变量不进 git。不用哈希是因为哈希不可逆，而 Key 调模型时需要还原明文，所以用可逆的对称加密。

**Q2（安全）：那密码呢？**
密码用 bcrypt 哈希——密码只需验证不需还原，用不可逆哈希更安全。「能还原的用加密、只验证的用哈希」。

**Q3（连接池）：连接池关键参数有哪些？pre_ping 干嘛？**
pool_size/max_overflow 控并发；pool_timeout 取连接超时；pool_recycle 防连接被单方面断开后变死连接；pre_ping 取连接前 ping 一下剔除失效连接，解决 DB 重启后首请求报错；statement_timeout 在 DB 层限单条 SQL 超时防慢查询挂死池。

**Q4（规范）：统一响应和异常怎么设计的？**
所有接口返回 {code,message,data}。业务失败抛 BizError（中文提示+业务码+HTTP码），全局异常处理器把 BizError、参数校验错、HTTP 错都转成统一结构，前端永远拿到可预测结构和中文提示，不暴露异常栈。

**Q5（进阶）：Fernet 底层是什么？安全吗？**
Fernet 是对称加密方案，底层 AES-128-CBC 加密 + HMAC-SHA256 认证 + 时间戳，能防篡改。安全性取决于密钥保管——密钥在 .env 不进 git、不打日志。

**Q6（进阶）：异步 SQLAlchemy 为什么 expire_on_commit=False？**
默认提交后对象属性过期，下次访问会触发新查询；异步下这会在不期望的地方触发 IO 甚至在 session 关闭后报错。设 False 让提交后对象仍可安全读取已加载属性。

## 7. 相关论文 / 概念

**① 加密 vs 哈希：可逆性决定选型**
密码学两类基础工具，本篇核心区分：
- **哈希（Hash）**：单向不可逆（明文→摘要，无法还原）。适合「只需验证不需还原」的场景——密码。**bcrypt（Provos & Mazières 1999）** 是专为密码设计的加盐自适应哈希，可调 cost 提高破解成本，抗彩虹表和暴力破解。
- **对称加密（Symmetric Encryption）**：可逆（同一密钥加解密）。适合「需要还原明文使用」的场景——API Key。**Fernet** 是 cryptography 库的对称加密规范，底层 **AES-128-CBC 加密 + HMAC-SHA256 认证 + 时间戳**，防篡改。
- **原则**：能还原的用加密、只验证的用哈希。本项目密码用 bcrypt、API Key 用 Fernet。

**② JWT（JSON Web Token，RFC 7519）与双令牌**
无状态鉴权标准：服务端签发带签名的 token，客户端每次带上，服务端验签即可，不用查 session 表（无状态、易水平扩展）。**access/refresh 双令牌**模式：access 短期（泄露损失小）、refresh 长期（免频繁登录），access 过期用 refresh 换新。

**③ 数据库连接池可靠性模式**
连接池是数据库高并发的标配，关键可靠性参数对应经典问题：
- **pre-ping**：取连接前先探活，解决「DB 重启/网络抖动后拿到失效连接、首次请求必报错」。
- **recycle**：连接定期回收，防被 DB 或中间件单方面断开后变「死连接」。
- **statement_timeout**：单条 SQL 超时，防慢查询挂死连接池。

**④ 统一响应 + 全局异常处理（API 设计规范）**
RESTful 实践：统一响应包络（`{code,message,data}`）让前端可预测、统一处理；全局异常处理器把各类异常（业务/校验/HTTP）转成统一结构，不向客户端暴露内部堆栈（既友好又安全）。

> 一句话脉络：密钥管理按「可逆性」选型（密码 bcrypt 哈希、Key Fernet 加密）；JWT 双令牌做无状态鉴权；连接池靠 pre-ping/recycle/超时保可靠；统一响应+全局异常保一致性和安全。

## 8. 可优化方向

- **密钥轮转**：Fernet 支持 MultiFernet 多密钥，可做 Key 轮换不停机。
- **连接池监控**：暴露 `get_pool_status()`（已实现）到监控面板，观测连接占用。
- **错误码集中管理**：业务码目前散落，可集中成枚举 + 文档，前后端共享。
- **限流/熔断**：接口层加限流，外部依赖（LLM/ES）加熔断降级。
