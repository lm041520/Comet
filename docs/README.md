| 06 工程化与部署 | 鉴权安全、异步任务、分享导出、推送、部署、跨模块踩坑 |

## 三、新增 / 补充文档怎么做

**新增一个功能的设计文档**：
1. 判断归属能力域（见上表），在对应文件夹新建 `NN-功能名.md`（**带两位数字前缀**，接在该域现有最大编号后，体现学习顺序）。
2. 套用「四、单篇模板」9 节结构。
3. **写前先读对应源码**（service/core/repository/prompt/model），以源码为准。
4. 在本文件「一、目录结构」里加一行登记。

> 编号约定：每个能力域内文档用 `01- 02- 03-` 前缀，按**合理的学习/依赖顺序**排（基础在前、应用/进阶在后）。`00/README.md` 作为模块入口不编号。

**补充到已有文档**（小功能/增强）：
- 如果是某篇的延伸（如给检索加个新策略），直接在那篇对应小节补，不单独成篇。
- 折叠原则：后端较薄或偏前端的小功能（如收藏、消息反馈），作为相关篇的子小节带过，不碎片化。

**改了已有功能**：同步更新对应文档的 §3 实现、§4 取舍、§5 踩坑（若有新坑）。

## 四、单篇文档模板（9 节，所有功能文档统一）

```markdown
# {功能名} — 设计与面试

> 一句话定位 + 对应 Agent 岗能力域。

## 0. 能力定位（对应招聘要求）   ← 对到 JD 哪条，一句话点题
## 1. 解决什么问题              ← 痛点 → 方案，先 why 再 how
## 2. 架构 / 数据流图           ← Mermaid，必有；配几句顺图讲
## 3. 核心设计与实现（后端）     ← 按执行顺序讲清机制+为什么，难点给「面试一句话」
## 4. 关键设计取舍（表格）       ← 方案A vs B 为什么这么选（重头）
## 5. 踩坑与解决                ← 现象→根因→解法
## 6. 面试问答（5~8题 由浅入深） ← 带参考答案（重头）
## 7. 相关论文 / 概念           ← 当知识点讲：发展脉络+概念详解+本项目怎么用
## 8. 可优化方向               ← 2~4 条体现 sense
```

**写作要求**：
- 只写后端（前端实现不进）；每篇必有 1 张 Mermaid 图。
- **§3 讲清机制和「为什么」但别过细**——流程化、概念化，关键函数点到为止，不堆变量类型/正则/`stack.enter_async_context` 这类代码级细节（过细反而难懂）。标杆：`04/三元组萃取`、`03/工具注册中心与MCP接入` 校准后的 §3。
- **§4 取舍表 + §6 面试问答是重头**。
- **§7 当知识点讲**：每个概念讲透（是什么/解决什么/思想），涉及论文梳理「早期→关键论文→演进」脉络，挂钩「本项目怎么用」。**只写确定真实的论文/年份，拿不准只讲思想不编引用**。
- 代码给路径、贴关键 5~15 行即可，靠文字讲清。
- 禁止出现参考来源/借鉴项目名（沿用编码规范）。

## 五、约定

- `_archive-release设计与八股/` 是按版本归档的历史快照（releasev0.0.3/4 各功能的后端八股），**不回改**；本套（00~07）是按能力域的长期主线，**持续更新**。
- 两者并存：版本快照看「某版本做了啥」，能力域看「某能力怎么做的」。
- 本目录是面试学习资料，`files/0 需求总结/` 是开发期进度台账（不进 git），分工不同。
